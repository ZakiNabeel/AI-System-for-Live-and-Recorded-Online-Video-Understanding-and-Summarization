# Plan 5.2 ŌĆö Domain-Specific Logic (Bonus / Stretch)

> **Self-contained scope.** Add a pluggable "domain mode" that re-targets the LLM prompts (Plan 4.2) and the output formatter (Plan 4.3) for a specific subject area: **education**, **trading**, **medical**, **law**, or **tutorial-strategy**. Optionally convert tutorial videos into pseudocode or executable workflow files. This plan covers the PRD's "Bonus Extensions" section.

---

## 1. Objective

Build `src/domain/` package that:

1. Defines a `DomainProfile` interface ŌĆö name, prompt overrides, output schema extensions, and post-processors.
2. Ships five built-in profiles, each implemented in its own file.
3. Adds a `--domain <name>` CLI flag that the pipeline (Plan 5.1) wires into Plans 4.2 and 4.3.
4. For the special `tutorial-strategy` profile, generates an extra deliverable: `strategy.py` (or `strategy.json`) ŌĆö a structured representation of the steps demonstrated in the video.

Everything in this plan is **additive**: with `--domain` unset, the project behaves exactly as before. No regression risk to the core pipeline.

---

## 2. Contract

### `DomainProfile`
```python
class DomainProfile(Protocol):
    name: str
    description: str

    def chunk_prompt_addendum(self) -> str: ...
    """Extra text appended to prompt_chunk.txt before the input."""

    def global_prompt_addendum(self) -> str: ...
    """Extra text appended to prompt_global.txt before the input."""

    def extra_output_schema(self) -> dict | None: ...
    """JSON schema fragment for fields the LLM should add (e.g.,
    `learning_objectives` for education). Return None if no extras."""

    def post_process(self, summary: dict, fused: dict, output_dir: Path) -> list[Path]: ...
    """Optional: generate domain-specific extra files. Return list of created paths."""
```

### Registry
```python
DOMAINS: dict[str, DomainProfile] = {
    "education": EducationProfile(),
    "trading":   TradingProfile(),
    "medical":   MedicalProfile(),
    "law":       LawProfile(),
    "tutorial-strategy": TutorialStrategyProfile(),
}
```

### Wiring into Plan 4.2
```python
def summarize(..., domain=None):
    profile = DOMAINS[domain] if domain else None
    chunk_prompt = base_chunk_prompt + (profile.chunk_prompt_addendum() if profile else "")
    global_prompt = base_global_prompt + (profile.global_prompt_addendum() if profile else "")
    ...
```

### Wiring into Plan 4.3
```python
def format_outputs(..., domain=None):
    ...
    if domain:
        extras = DOMAINS[domain].post_process(summary, fused, output_dir)
        result.extras = extras
```

## 3. Profile Designs

### 3.1 `education`
**Goal:** Lecture / tutorial / class.

**Prompt addendum (chunk):**
> Additionally extract:
> - `learning_objectives`: 3ŌĆō5 bullet points of what the student should learn.
> - `worked_examples`: list of `{timestamp, problem_statement, key_steps, final_answer}`.
> - `definitions`: list of `{term, definition, timestamp}` for new vocabulary introduced.

**Output extras:** `study_notes.md` ŌĆö a re-organized "study guide" with definitions glossary, examples worked, and practice questions extrapolated from the content.

### 3.2 `trading`
**Goal:** Trading / financial-markets analysis videos.

**Prompt addendum:**
> Additionally extract:
> - `tickers_mentioned`: list of ticker symbols ($AAPL, BTC, etc.).
> - `signals`: list of `{timestamp, ticker, action: "buy|sell|hold|watch", rationale, timeframe}`.
> - `indicators`: technical indicators discussed (RSI, MACD, etc.) with timestamps.
> - `risk_warnings`: any caveats stated.

**Output extras:** `trade_log.csv` ŌĆö `timestamp, ticker, action, rationale`.

**Disclaimer banner** automatically prepended to all outputs in this domain: "This summary is generated by AI for informational purposes only and is not financial advice."

### 3.3 `medical`
**Goal:** Medical-education videos, conference talks, lectures.

**Prompt addendum:**
> Additionally extract:
> - `conditions_discussed`: ICD-10-style brief labels.
> - `treatments_mentioned`: drug or procedure names with timestamps.
> - `evidence_levels`: cite any RCTs, meta-analyses, or guidelines mentioned.
> - `key_clinical_points`: actionable takeaways for clinicians.

**Disclaimer banner**: "Educational summary only. Not for diagnostic or treatment decisions. Always verify with primary sources."

**Output extras:** `clinical_notes.md` ŌĆö formatted with subjective/objective/assessment/plan-style sections where applicable.

### 3.4 `law`
**Goal:** Legal lectures, depositions, case discussions.

**Prompt addendum:**
> Additionally extract:
> - `cases_cited`: `{case_name, citation, jurisdiction, timestamp, relevance}`.
> - `statutes`: any statutory references.
> - `legal_principles`: doctrines or rules discussed.
> - `arguments`: pro/con structure if a debate.

**Output extras:** `case_brief.md` ŌĆö IRAC-style brief (Issue, Rule, Application, Conclusion) when content fits.

**Disclaimer:** "Not legal advice. Generated by AI; verify all citations independently."

### 3.5 `tutorial-strategy` (the headline bonus)
**Goal:** Convert how-to tutorials (cooking, programming, software walk-throughs, gameplay strategies) into machine-readable steps.

**Prompt addendum:**
> Additionally produce:
> - `task`: one-sentence description of what the tutorial teaches.
> - `prerequisites`: list of strings.
> - `steps`: ordered list of `{step_number, timestamp, action, parameters, expected_outcome, screenshot_frame_index}`.
> - `decision_points`: `{step_number, condition, branch_yes, branch_no}` for "if X then Y" branches.

**Output extras:**
1. **`strategy.json`** ŌĆö the structured workflow.
2. **`strategy.md`** ŌĆö human-readable numbered list with embedded screenshots.
3. **`strategy.py`** (optional, only for programming tutorials) ŌĆö a Python skeleton with one function per step:
    ```python
    # Auto-generated from <video_url>
    # Task: <task>

    def step_01_install_dependencies():
        """[00:32] Run `pip install requests`."""
        # TODO: implement
        ...
    ```
4. **`strategy.bpmn.xml`** (optional, future) ŌĆö BPMN representation of branched workflows.

## 4. Phased Implementation

### Phase A ŌĆö Profile interface & registry (~30 min)
Create `src/domain/__init__.py`, `src/domain/base.py` with the `DomainProfile` Protocol, and `src/domain/registry.py` exposing `DOMAINS`.

### Phase B ŌĆö Each profile (~1 hr ├Ś 5)
Implement one file per profile (`education.py`, `trading.py`, etc.). Most profiles only need to provide:
- `chunk_prompt_addendum()` returning a string.
- `post_process()` writing one or two files derived from `summary` and `fused`.

The prompt strings live in `src/llm/prompts/domains/<name>.txt` (already referenced in Plan 4.2). The Python class loads its addendum from there:
```python
class EducationProfile:
    name = "education"
    description = "Lecture and educational-content profile."
    def chunk_prompt_addendum(self):
        return (Path(__file__).parent.parent / "llm" / "prompts" / "domains" / "education.txt").read_text()
    ...
```

### Phase C ŌĆö Tutorial-strategy profile deep dive (~3 hr)
This is the most complex profile. Implementation notes:

1. **Step extraction** ŌĆö the LLM produces `steps[]` already; the profile's post_process simply reformats.
2. **Pseudocode generation** ŌĆö only when `task` mentions programming keywords (`python`, `function`, `code`, `script`, `command`, `git`, `docker`, etc.). Detect via a small keyword list. If detected, render `strategy.py`:
   ```python
   def render_python_skeleton(steps, task, url):
       header = f'"""Auto-generated workflow for: {task}\nSource: {url}\n"""\n\n'
       fns = []
       for s in steps:
           name = f"step_{s['step_number']:02d}_" + slugify(s['action'])[:40]
           ts = fmt_time(s['timestamp'])
           doc = f'    """[{ts}] {s["action"]}.\n\n    Expected: {s["expected_outcome"]}\n    """'
           body = "    # TODO: implement\n    ..."
           fns.append(f"def {name}():\n{doc}\n{body}\n")
       return header + "\n\n".join(fns) + "\n\n\nif __name__ == '__main__':\n    " + \
              "\n    ".join(f"step_{s['step_number']:02d}_*()" for s in steps)
   ```
3. **Validation** ŌĆö run `ast.parse(rendered_code)`; if it fails, fall back to embedding the steps as comments inside a single function.
4. **Branched workflows** ŌĆö if `decision_points` non-empty, render as `if/else` blocks; else linear.

### Phase D ŌĆö Wire into Plan 4.2 (~30 min)
Already covered in Plan 4.2's `domain` parameter. Verify the addendum text is included and the LLM extra fields show up in `summary.raw.json` under a `domain_extras` key.

### Phase E ŌĆö Wire into Plan 4.3 (~30 min)
Add a `--domain` flag to formatter CLI. After main outputs, call `profile.post_process(...)`. Append the extra files to `report_card.md`.

### Phase F ŌĆö Wire into Plan 5.1 (~15 min)
Pipeline already accepts `domain` and forwards. Document that `--domain tutorial-strategy` adds `strategy.json`/`strategy.md` to the deliverables.

### Phase G ŌĆö Tests (~2 hr)

1. **Profile registry** ŌĆö assert all five names resolvable; unknown name ŌåÆ `UnknownDomainError`.
2. **Prompt addendum applied** ŌĆö call `summarize` with a fake provider; verify the prompt text passed to the provider contains the addendum.
3. **Education `study_notes.md`** ŌĆö feed canned summary + fused; assert file exists and contains "Definitions" and "Worked Examples" headings if those fields are in the input.
4. **Trading `trade_log.csv`** ŌĆö assert CSV has correct headers and one row per signal.
5. **Tutorial-strategy `strategy.py`** ŌĆö feed canned steps; assert resulting `.py` parses (`ast.parse`) and has correct number of functions.
6. **Disclaimer banners** ŌĆö for `medical`/`law`/`trading`, assert `summary.md` first 200 chars include the disclaimer string.
7. **No-domain regression** ŌĆö run full pipeline without `--domain`; assert no domain-specific files created.

## 7. Edge Cases & Pitfalls

1. **LLM doesn't return `domain_extras`** ŌĆö model occasionally ignores the addendum. Re-prompt once with stricter instructions; on second failure, log a warning and proceed with empty extras.
2. **Trading tickers false positives** ŌĆö the model may extract `$5` as a "ticker." Filter to uppercase 1ŌĆō5 letter strings, optionally followed by exchange suffix (`.NS`, `.L`, etc.).
3. **Medical drug/treatment names misspelled** ŌĆö out of scope to validate; surface verbatim with the disclaimer.
4. **Legal case citations** ŌĆö formats vary widely. Don't try to validate. Just extract verbatim.
5. **Tutorial-strategy on non-tutorial videos** ŌĆö the `steps` list may be near-empty. Fall back to producing a single-step strategy. Don't crash.
6. **Pseudocode generator on non-programming tutorials** ŌĆö skip `strategy.py`; only produce `strategy.md`.
7. **Profile-specific token budget** ŌĆö addenda increase prompt size; be mindful of model context limits. Plan 4.1's chunker already handles this, but document.
8. **Extending profiles** ŌĆö keep the interface small (Protocol with 4 methods). Anyone can add a new profile by dropping a file in `src/domain/` and registering it.

## 8. Out of Scope

- Profiles for languages other than English.
- Auto-detecting which profile to use (could be a future enhancement: a tiny classifier on the title + first-minute transcript).
- Executing the generated `strategy.py` (the file is a skeleton; users implement the bodies).
- Compliance certifications for medical/legal use (clearly disclaimed).

---

## 9. Definition of Done

A developer can run `python -m src.pipeline --url <programming_tutorial_url> --domain tutorial-strategy` and find, in `data/output/<run_id>/`, both the standard deliverables and a `strategy.py` / `strategy.md` describing the tutorial as ordered, callable steps ŌĆö using only this plan file and the prior plans' outputs.
